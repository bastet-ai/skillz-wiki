## Operator triage

1. **Map who controls each trigger.** Distinguish anonymous public callers, authenticated builders, organization owners, repository contributors, low-privilege app users, IdP provisioning clients, CSV importers, and holders of leaked database backups.
2. **Separate validation-time from use-time.** For SSRF, redirects, mirror URLs, ZIP paths, and URL helpers, record whether allowlists run before or after normalization, redirect following, DNS lookup, symlink resolution, and final request construction.
3. **Use canary-only proofs.** Disposable buckets, owned redirectors and DNS names, fake chat identities, scratch repositories, harmless attachments, marker notebooks, synthetic LFS blobs, inert branch-name commands, marker Git protocol pushes, inert webhook events, synthetic SCIM users, CSV rows, stale-session status checks, and fake enrollment tokens are sufficient.
4. **Avoid sensitive endpoints.** Do not target cloud metadata, production loopback panels, real datasource buckets, real chat accounts, real skill libraries, live workflow side effects, private customer attachments, financial exports, or production enrollment tokens.
5. **Preserve role and topology evidence.** Strong reports show route family, required auth role, datasource/workspace visibility, redirect or DNS-rebind chain, archive path resolution, process owner, and patched-vs-vulnerable comparisons.

## Replayable validation boundaries

### Budibase datasource, chat identity, and OAuth2 harness

- Preconditions: owned Budibase lab or explicit customer scope, disposable workspace, fake S3-compatible bucket, owned chat workspace identities, and one low-privilege builder/user account.
- For signed uploads, create a synthetic S3 datasource with throwaway credentials, then request a presigned URL from a session that should lack datasource access. Upload only an inert marker object to a disposable bucket/key.
- For chat identity handoff, create the attacker-side link token, then visit the handoff URL from a separate disposable authenticated victim browser profile. Evidence is the resulting mapping between two lab identities.
- For OAuth2 SSRF, submit token endpoint URLs pointing to owned callback collectors and owned redirectors. Capture callback timing and validation-envelope behavior only.
- For DNS-rebinding SSRF, configure an owned hostname that returns an allowed public IP for the validation lookup and a blocked lab canary IP for the later connection. Keep the second target under your control; do not use metadata, loopback admin panels, or production RFC1918 services.
- For public webhook trigger mass assignment, seed two disposable workspaces and a webhook whose only side effect is writing a marker to a canary table. Send a body containing an `appId` collision and record whether execution crosses workspace context.
- For PWA ZIP imports, include a symlink entry pointing to a synthetic outside-root marker and list that symlink path in `icons.json`; evidence is whether the served asset contains the marker bytes. Never point symlinks at `.env`, database files, cloud credentials, or user uploads.
- Negative controls: authentication and builder/table permission checks on attachment routes, CSRF/consent on chat-link handoff, URL allowlists applied after redirects, and no response-body leakage from failed OAuth2 validation.

### Gogs webhook redirect SSRF harness

- Preconditions: scoped Gogs instance, disposable repository, permission to create/test webhooks, and an owned redirector domain.
- Create a webhook pointing to the owned redirector. Make the redirector return a 30x to a second owned URL that simulates a blocked host class without using real metadata or internal services.
- Positive evidence is the Gogs server following the redirect and reaching the second canary despite creation-time URL checks.
- Negative controls: redirect disabled or manually handled, final URL host/IP revalidated after every redirect, DNS rebinding resistance, and egress segmentation.

### Gogs local import, organization, attachment, and notebook harness

- Preconditions: scoped Gogs lab, disposable organization, attacker/victim lab users, private scratch repositories, harmless attachments, and local scratch repositories containing marker commits only.
- Compare new-migration validation against mirror-settings edits with `file://` or local-path repository values that should be blocked. Positive evidence is import of the marker repository through mirror settings when the new-migration path rejects it.
- For organization CSRF, create a disposable Owners team and test whether a GET request from the victim-owner browser adds the attacker user. Evidence is a role-change table for lab accounts only.
- For attachment authorization, upload a harmless marker attachment to a private repo, then request `/attachments/:uuid` as unauthenticated and unauthorized lab users. Capture status, content hash of the marker, and parent-object permissions.
- For notebook previews, commit an `.ipynb` whose Markdown contains a harmless click-only DOM marker link. Capture whether client-side re-rendering recreates a dangerous scheme after server sanitization.
- Negative controls: identical local-import validation on create and edit paths, CSRF tokens on all state-changing methods, parent-object authorization before attachment file open, and no client re-render after sanitized notebook insertion.

### Gogs Git forge tenant, protocol, and merge harness

- Preconditions: isolated Gogs lab, disposable orgs/repos, attacker/victim lab users, a synthetic Git HTTP server, a synthetic LFS object corpus, and temporary filesystem marker directories owned by the lab process.
- For organization-team API disclosure, request `GET /api/v1/orgs/:orgname/teams` as unauthenticated and low-privilege users; record only route, status, and synthetic team metadata exposure.
- For org-name path traversal, create only scratch organizations and repositories whose resolved paths point at disposable temp directories. Evidence is path resolution or marker repository placement; do not overwrite real hooks, SSH keys, config, or service files.
- For LFS dedupe, seed a private lab repo with a harmless LFS marker blob, then attempt to bind the same OID from a second repo without uploading matching bytes. Evidence is whether the second repo can fetch the marker through its own authorization context.
- For repository file uploads, commit a parent-directory symlink to a disposable target, then upload a marker filename that exercises backslash/path cleanup. Capture whether the marker appears outside the work tree.
- For Git smart HTTP confusion, grant a collaborator read-only access, then send a receive-pack request whose query string claims `service=git-upload-pack`. Evidence is acceptance or rejection of a marker commit to a scratch repo.
- For repository settings API drift, compare write-level collaborator calls against UI-visible admin-only controls for issue tracker, wiki, and mirror-sync settings. Keep changes reversible and scoped to throwaway repos.
- For milestone DOM sinks, create a milestone with a harmless visible marker and test each issue page/dropdown that renders it; do not include credential theft or privileged action payloads.
- For rebase-merge command injection, use a lab-only branch/refname that can run an inert marker command if accepted, such as writing to a temp file. Never run shell payloads on shared or customer-hosted Gogs.
- For migration redirect SSRF, submit an owned public Git URL that redirects to a synthetic blocked-host Git server you control. Evidence is redirect following and import attempt metadata, not internal repository contents.
- For collaboration owner escalation, submit raw access-mode values from a repo-admin lab account and record role transitions for disposable users only.
- For `redirect_to`, use `example.invalid` or an owned benign destination to prove browser-normalized off-origin navigation from slash/backslash traversal inputs.
- Negative controls: namespace validation before filesystem writes, every path component checked for symlinks, LFS body hash verified even on dedupe hits, authorization keyed to the actual Git RPC endpoint, API routes requiring the same role as UI routes, branch names passed after `--` or rejected before shell/Git execution, final migration URL validation after redirects, owner mode unreachable by collaborator APIs, and URL validation after browser-equivalent normalization.

### skillctl hostile library harness

- Preconditions: scratch project, scratch skill library, fake outside-root marker files, and no real credentials or home-directory paths.
- Test `.skills.toml` `source_sha` values that look like Git flags and confirm whether the tool rejects them before invoking `git ls-tree`.
- Add FIFO, device, socket, and hardlink canaries to a malicious skill folder and verify the copy path rejects non-regular files instead of blocking, reading unbounded data, or preserving hardlink relationships.
- Exercise `add --dest` only against temp directories and confirm absolute paths or `..` traversal cannot delete or replace outside-root markers.
- Capture command, version, resolved paths, before/after trees, and rejection messages.

### Automation and identity-parser harnesses

- For n8n, send one unsigned fake GitHub webhook and one properly signed canary event to a lab workflow whose only action writes a harmless marker. Positive evidence is workflow execution without valid `X-Hub-Signature-256`.
- For Nuxt, build a route table covering `/..//host`, `/.//host`, encoded-dot variants, protocol-relative paths, and `javascript:` targets passed through `navigateTo(..., { open })`; capture only redirect locations or blocked decisions.
- For `scim-patch`, apply a PATCH with `__proto__.skillz_canary=true` to a disposable SCIM user in a same-process lab and verify whether unrelated plain objects inherit the marker.
- For nebula-mesh, seed a lab SQLite DB with a pending synthetic enrollment token and show whether reading the DB reveals a consumable raw token; stop before touching real overlay identities.

### Actual CSV and stale-session harness

- Preconditions: owned Actual lab, OpenID multi-user mode when testing session revocation, disposable transactions, and two lab users.
- For CSV formula injection, import or edit transaction `Payee`, `Notes`, `Account`, or `Category` fields with inert spreadsheet formulas that reference only owned callback URLs or local markers, then export CSV and confirm whether formula metacharacters survive as the first character of a cell.
